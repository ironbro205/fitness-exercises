# 헬스앱 배포(출시)

헬스앱은 **Vercel 자동 배포**다 — `main`에 변경이 병합되면 Vercel이 알아서 웹에 올린다.
즉 **배포 = main 병합**. 하지만 그것만으로는 부족하다. 아래 캐시 버전 단계가 핵심이다.

## 왜 캐시 버전을 올려야 하나 (가장 중요)

`service-worker.js`는 `index.html`을 포함한 자원을 `CACHE_VERSION`(예: `health-app-v12`) 이름으로 캐싱한다.
**캐시 이름이 그대로면, 사용자 폰은 새 코드가 올라가도 옛날 코드를 계속 쓴다.** 그래서 코드가 바뀌면 반드시 버전을 올려 사용자가 새 버전을 받게 해야 한다.

## 절차

1. **캐시 버전 올리기** — `service-worker.js`에서 `CACHE_VERSION`의 숫자를 +1 한다 (`v12` → `v13`).
2. **새 정적 파일 연결 확인** — 이번 변경에서 `js/*.js`나 `css/*.css` 같은 새 파일을 추가했다면:
   - `index.html`에 `<script>`/`<link>`로 연결돼 있는지
   - `service-worker.js`의 `CORE_ASSETS` 목록에 들어갔는지
   둘 다 확인한다. (Network-First라 안 넣어도 동작은 하지만, 오프라인 보장을 위해 넣는다.)
3. **QA 통과 확인** — `.claude/QA_CHECKLIST.md`의 "항상 확인" 항목을 통과했는지 본다. 통과 못 했으면 배포하지 말고 먼저 고친다.
4. **커밋 → PR → 병합** — 기본 브랜치(`main`)에 직접 커밋하지 말고:
   - 작업 브랜치를 만들고 커밋 (커밋 메시지 끝에 Co-Authored-By 트레일러)
   - `gh`로 PR 생성 (`gh pr create`)
   - 사용자 승인 후 `main` 병합 → 이 순간 Vercel이 자동 배포 시작
   > 코드 변경이면 병합 전에 Codex 리뷰가 끝나 있어야 한다(전역 규칙). 이 스킬은 이미 리뷰·QA를 통과한 변경을 "내보내는" 단계다.
5. **라이브 확인** — 배포가 끝나면 실제 주소를 폰/브라우저에서 **강력 새로고침**으로 열어, 새 버전이 보이고 정상 동작하는지 확인한다. (서비스워커가 새 캐시로 교체되며 한 번 더 새로고침이 필요할 수 있다.)

## 비개발자용 한 줄 요약

"코드 고침 → 캐시 버전 +1 → PR 만들어 병합 → Vercel이 자동으로 올림 → 폰에서 새로고침해 확인." 이 순서를 빼먹지 않는 게 전부다.

## 점검 신호

- 사용자가 "분명 고쳤는데 폰에선 그대로야"라고 하면 → 십중팔구 캐시 버전을 안 올렸거나 강력 새로고침을 안 한 것이다.
